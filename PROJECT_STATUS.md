## 问卷调研关键结论（87份反馈，约一半为物理系学生）

### 现状与痛点
- 88%学生遇到不懂问题首选用AI（ChatGPT/Gemini等）
- 对现有AI最不满：瞎编不可信(58)、缺公式图表(37)、回答太泛不针对课程(27)、直接给答案没学会(21)、担心上传课件泄露(6)
- 最大学习困难：概念抽象(43)、理解了但不会做题(33)、公式混淆(33)、缺复习资料(31)、反馈滞后(29)

### 产品形态（强信号）
- **网页版是首选**（54票，远超手机端29/桌面端20/嵌入微信4）→ MVP必须是网页应用
- 设备习惯：电脑(64) > 平板(46) > 手机(26)

### 引导式学习（核心卖点已验证）
- 79/86认为引导式学习有帮助；1-10分平均意愿7.5分
- 最受欢迎方式（57票）："先帮我理清思路，等我说想看答案/尝试后再给答案"
- **重要：不能一刀切**——用户着急时（如考前/赶作业）需要能直接给答案，引导模式应可调节/场景感知，而非强制苏格拉底式
- 最需要场景：考前复习(64)、做作业遇难题(53)、预习(38)

### 附加功能期望（优先级供后续参考）

## 技术路线（已确认）

| 模块 | 选择 | 备注 |
|---|---|---|
| 知识库内容管理 | **Obsidian（免费）+ GitHub同步** | 个人/非商业用途免费，无需Obsidian Sync付费服务 |
| 向量库 | Chroma（本地Python库，非独立服务） | 免费，MVP够用 |
| Embedding模型 | 开源多语言模型（如 bge-small-zh / multilingual-e5），本地运行 | 中英文课件都能处理，零成本 |
| 后端 | Python + FastAPI | 一开始就写规范API，前端无论如何升级都不浪费 |
| LLM | **DeepSeek API（deepseek-v4-pro）** | 1M输入(cache miss) $0.435 / 1M输出 $0.87；自带thinking mode（默认开）。单次问答(~2000输入+500输出,cache miss)约$0.0013，团队5人×20次/天×30天≈$3.9 |
| 前端 | **Next.js + FastAPI（前后端分离，一步到位）** | 避免后期从Streamlit/Chainlit重写；MVP阶段功能保持最简 |
| 部署 | 本地开发先跑通，后续Render免费层 | 先验证逻辑 |

公式渲染：KaTeX/MathJax。图表识别等高级功能放到三阶段。

## MVP范围

**做（最小循环）**：
1. 选1-2门资料最全课程（量子力学/经典力学/电磁学票数最高）作为试点
2. 团队整理课程资料为Markdown
3. 脚本：切块 → embedding → 存入Chroma
4. 后端API：问题 → 检索 → 拼接prompt（含场景感知的引导逻辑）→ DeepSeek生成 → 返回回答+引用来源
5. 前端：聊天界面，支持公式渲染
6. 流式输出（Streaming）：后端通过 SSE 逐步推送 LLM 生成内容，前端实时逐字显示，避免长时间空白等待。物理问题回答通常较长，流式输出是可用性基线
7. 本地对话记录保存，每当重新进入项目网页，自动打开最近一次对话框，可打开任意一次对话记录并继续互动，也可新建对话框开启新的话题

**暂不做**：用户系统、知识体系图谱、错题集、社区、匿名问题收集、移动端、付费功能（留给三/四阶段）

- **多模型切换（V4-Pro + R1）**：根据问卷"引导需场景感知"的发现，设想在"烧脑解题"场景切换到推理能力更强的模型（如DeepSeek-R1）。

  - R1价格暂未核实，待需要时再查

- **多模态（图片/公式识别）**`【技术方案已确认 2026-06-22】`：
  
  **使用场景**（两个）：
  1. 用户拍摄/上传带公式的照片（手写体或电子字体）
  2. 用户上传带公式的 PDF（作业、考试真题、讲义）
  
  **技术栈选型**：

  | 模块 | 选型 | 理由 |
  |---|---|---|
  | 照片公式 OCR | **Pix2Text**（`breezedeus/Pix2Text`）+ **UniMERNet** 后端 | 开源最高精度组合；中英文混合识别；支持手写+电子字体；一张图多公式自动定位分别识别 |
  | PDF 解析 | **MinerU**（`opendatalab/MinerU`） | UniMERNet 同团队出品，专为学术文档设计，公式+文字+表格一体化提取，输出 Markdown+LaTeX |
  | 前端公式编辑器 | **MathLive**（`mathlive` npm 包） | 开源可视化 LaTeX 编辑器，自带数学符号虚拟键盘，支持触屏，LaTeX 输入输出直接对接 KaTeX |
  
  **核心交互流程**（公式确认环节）：
  ```
  用户上传照片/PDF → 后端 OCR 识别 → 返回 LaTeX
       ↓
  前端将 LaTeX 预填入 MathLive 编辑器，展示渲染后的公式
       ↓
  用户确认："请确认识别结果是否正确"
    - [✓ 确认正确] → LaTeX + 用户提示词 → /chat API → LLM 回答
    - [编辑修改] → 用户在 MathLive 中逐元素编辑（可用符号面板）→ 改完后点确认
    - [✗ 重新上传] → 返回上传界面
  ```
  - 物理学科特殊符号面板需定制：bra-ket 记号、偏导算符、狄拉克符号、张量积等
  - MathLive 替代了原始"逐元素 bounding box"方案（后者需自建目标检测+交互系统，工时 4-6 周，不可行）

  **分阶段实施路线**：
  
  | 阶段 | 内容 | 预估工时 |
  |---|---|---|
  | Phase 1（三阶段前期） | 后端集成 Pix2Text + `/ocr/image` 端点；前端添加图片上传按钮；OCR 结果直接填入聊天框（最简版） | 1-2 天 |
  | Phase 2（三阶段中期） | 集成 MathLive 编辑器 + 确认/编辑/取消交互流程；定制物理符号面板 | 3-4 天 |
  | Phase 3（三阶段后期） | 后端集成 MinerU + `/ocr/pdf` 端点；多页 PDF 处理；每个公式可独立编辑确认 | 2-3 天 |
  
  **部署注意**：Pix2Text + UniMERNet 约 1-1.5GB 模型体积，建议 GPU 推理；Render 免费层可能不够，需评估付费实例或自建服务器

### 已知问题 / 待改进
- ⚠️ **检索质量**：当前用 Chroma 默认向量模型（all-MiniLM-L6-v2，英文为主），对中文物理内容检索精度差（测试中"无限深势阱"问题没能召回对应段落）。**建议升级为多语言向量模型**：用 `fastembed`（基于 onnxruntime，无需 torch，轻量、可放 E 盘）加载 `bge-small-zh` 或 `multilingual-e5`。这是 MVP 跑通后的首要质量改进。
- AI回答的引用文献source目前无法点击后打开查阅
- 一旦AI的回复内容过长，由于一键切换中英文的按钮在回复框最上面，就会造成滑页不方便的情况。可要是把一件翻译按钮置顶，那么用户很难辨认出翻译按钮到底在对哪个回复框进行操作，导致交互不便。
